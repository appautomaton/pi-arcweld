Local Pi workspace / 01

Curate Pi locally.
Keep upstream clean.

pi-arcweld keeps a curated Pi agent environment in one auditable workspace: user-level extensions, system-instruction guidance, MCP tooling, and a runtime built outside the upstream checkout.

Repository boundary / 02

One upstream. A deliberate local layer.

Pi remains upstream. pi-arcweld pins pi-mono/ as a Git submodule and keeps its own work where it belongs: extensions in extensions/, response guidance in system-instruction/, and runtime maintenance in scripts/.

That boundary keeps experimentation precise. Upstream source stays lean; the local runtime is assembled and verified elsewhere; custom behavior has an obvious home.

Components / 03

Local changes with a defined edge.

Each component has one defined role. Together, they provide a maintainable Pi environment without creating an upstream fork.

  1. Component / 01Runtime

    Build outside the checkout.

    Assemble a local Pi runtime without turning the upstream source tree into a private build environment.

    • Package output and runtime state stay under build/.
    • The helper smoke-checks the runtime before promotion.
    • The user command can point at the verified local artifact.
    Runtime workflow
  2. Component / 02Plan mode

    Plan without cache churn.

    Plan state is appended instead of reshaping Pi’s active tools or system prompt, preserving the provider’s cached prefix through a plan-mode transition.

    • Built-in write and edit calls are restricted to canonical plan files.
    • Model calls to Pi’s built-in Bash are blocked while active.
    • It is a guard for local mutation, not a universal sandbox.
    Plan mode notes
  3. Component / 03MCP client

    MCP with deliberate boundaries.

    A small client discovers user-configured capabilities in the background while presenting the model a fixed, bounded tool surface.

    • User-global configuration is the explicit consent boundary.
    • Secret values stay private and missing variables fail closed.
    • Server metadata and output are treated as untrusted.
    MCP security model
  4. Component / 04Clarify

    Give decisions a shape.

    A focused, keyboard-operable question interface turns ambiguous choices into visible options, explicit answers, and a clear path for “other.”

    • Supports single and multi-select questions.
    • Moves through multi-question flows without losing context.
    • Keeps cancellation and free-text answers intentional.
    Clarification UI
  5. Component / 05Instruction layer

    Refine the response, not Pi.

    A global append layer improves response quality without replacing Pi’s generated system prompt or freezing its evolving tool context.

    • Preserves the runtime-generated prompt as the primary system head.
    • Layers behavior corrections on top of live tool context.
    • Lives globally rather than shadowing projects by accident.
    Instruction workspace

Set up the workspace / 04

Build the local Pi environment.

Clone the workspace with its pinned upstream source, build the external runtime, and verify the user-level Pi command.

bootstrap / local machine 
# clone the workspace and pinned Pi source
$ git clone --recurse-submodules https://github.com/appautomaton/pi-arcweld.git pi-arcweld
$ cd pi-arcweld

# assemble and verify the external runtime
$ scripts/build-pi-agent.sh --link-user-bin
$ pi --version

The root intentionally has no shared npm workspace. Local packages own their dependencies; upstream Pi keeps its own build and release process. Use the update helper when Pi moves.

Operating principle / 05

Pi stays upstream. Arcweld stays intentional.

The workspace stays maintainable when every curated component has an explicit owner, loading path, and verification boundary.
